## 四、数据流全链路

```
Bilibili XML 文件
    │
    ▼
BilibiliXmlParser.parse() → List<DanmakuItem>
    │
    ▼
DanmakuRepository.load() → DanmakuTimeIndex (1000ms buckets)
    │
    ▼
MpvClockBridge (Choreographer 每帧回调)
    │
    ├─ PlayerClockProvider.currentPositionMs()  ← MpvPlayerManager
    │
    ▼
DanmakuTimelineEngine.onClockTick()
    │
    ├─ Repository.query() → 时间窗口内弹幕
    ├─ TrackAllocator.allocateScroll/Fixed() → 分配轨道
    ├─ ActiveDanmakuStore → 维护活动弹幕
    │
    ▼
engine.snapshot() → List<DanmakuRenderItem>
    │
    ▼
CanvasRenderer.render()
    │
    ├─ GlyphCache.draw() → 预烘焙 Bitmap
    ├─ lockCanvas / drawBitmap / unlockCanvas
    │
    ▼
SimpleOverlayHost (透明 SurfaceView) → 显示在视频之上
```

### 13.2 解析链路验证

对真实哔哩哔哩 XML 文件增加了解析统计日志，确认当前“弹幕显示数量少”**不是解析器漏解析**。

实际验证结果：

| 指标 | 值 |
|---|---|
| XML 总节点数 | 9492 |
| 成功解析数 | 9492 |
| 空文本 / 空属性丢弃 | 0 |
| p 属性字段不足 | 0 |
| 时间字段非法 | 0 |
| 类型分布 | 1=7681, 5=1811 |

结论：

1. 当前 Bilibili XML 解析器未出现大规模漏解析
2. 显示数量问题主要来自时间轴调度、轨道容量和密度裁剪，而不是 parser 层
3. 当前测试样本以滚动弹幕和顶部固定弹幕为主，未观察到底部固定弹幕数据占比

### 13.5 当前已接受的工程取舍

为优先满足当前联调验收，本轮对密度策略做了偏实用的取舍：

1. 默认密度固定为最高档（`Crowded`）
2. `maxVisibleCount` 提高到 180
3. 滚动轨道数量按当前屏幕可容纳的最大安全值开放

这意味着当前版本更偏向“先尽量显示更多弹幕”，后续如果要恢复严格意义上的三档密度体验，需要再单独回调策略。
